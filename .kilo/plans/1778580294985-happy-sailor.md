### 4.2 Locking Mechanism for Leave Processing
**File**: Update `leaves/views.py` and `leaves/models.py`
**Logic**:
- When handler opens a leave in "ΠΡΟΣ ΕΠΕΞΕΡΓΑΣΙΑ" status:
  - If not locked: lock it to this handler (set locking_user, locked_at)
  - If locked by same handler: allow continue
  - If locked by different handler: show "Locked by X" message, offer manual unlock
  - Auto-unlock after 30 minutes of inactivity (check in view, not celery for simplicity)
- Show "Υπό επεξεργασία από [handler name]" to other handlers
- Unlock when handler saves/completes/rejects

### 4.3 Revocation System
**File**: Update `leaves/models.py` LeaveRequest methods

**Two types of revocation**:
1. **Ανάκληση Αίτησης** (until ΠΡΟΣ ΕΠΕΞΕΡΓΑΣΙΑ):
   - Available from ΚΑΤΑΧΩΡΗΘΗΚΕ through ΕΓΚΡΙΣΗ ΑΠΟ ΠΡΟΪΣΤΑΜΕΝΟ
   - User clicks button → status changes to ΑΝΑΚΛΗΣΗ ΑΙΤΗΣΗΣ ΑΠΟ ΑΙΤΟΥΝΤΑ
   - No leave days deducted
   - Terminal state

2. **Ανάκληση Ολοκληρωμένης Άδειας** (from ΟΛΟΚΛΗΡΩΜΕΝΗ):
   - User selects completed leave → clicks "Ανάκληση Ολοκληρωμένης"
   - Creates NEW LeaveRequest with `parent_leave` pointing to original
   - Status starts at ΚΑΤΑΧΩΡΗΘΗΚΕ, goes through full workflow
   - PDF includes reference to original decision
   - Original leave status changes to ΑΝΑΚΛΗΣΗ ΟΛΟΚΛΗΡΩΜΕΝΗΣ ΑΔΕΙΑΣ
   - Leave days are restored to user's balance

3. **Μερική Ανάκληση**:
   - Same as above but only for specific days
   - Create new LeaveRequest with `parent_leave`
   - Track revoked_days on original
   - Recalculate remaining balance

4. **Delete in ΠΡΟΣ ΕΠΕΞΕΡΓΑΣΙΑ** (handler only):
   - Only handler can delete
   - Log the deletion
   - Restore leave days to user's balance

### 4.9 SDEY Manager Restrictions
**File**: Update `accounts/models.py` User model and views
**Logic**:
- SDEY managers (users with `department.department_type.code == 'SDEY'`) have `can_request_leave = False`
- They can ONLY view dates and status of their SDEY's leave requests
- Cannot view attachments
- Cannot approve
- Cannot request leave for themselves
- Update all views to check `can_request_leave` before allowing request
**Implementation**:
- Override `can_request_leave` property in User model:
```python
@property
def can_request_leave(self):
    # SDEY managers can never request leave
    if self.department and self.department.department_type and \
       self.department.department_type.code == 'SDEY':
        return False
    # Root department managers (e.g., PDEDE director) cannot request
    if self.department and self.is_manager_of_department(self.department):
        if not self.department.parent_department:
            return False
    return True
```
- In all views that create leave requests, check in `dispatch()`:
```python
if not self.request.user.can_request_leave():
    raise PermissionDenied("Δεν έχετε δικαίωμα υποβολής αιτήσεων άδειας.")
```
- In templates, hide "Νέα Αίτηση" button: `{% if user.can_request_leave %}`
- On registration approval, auto-set `can_request_leave = False` for SDEY users

## PHASE 11: FSM Redesign - Καθαρό Workflow με Django FSM

### Βασική Αρχή
Αντί να αποθηκεύουμε UI states (Προεπισκόπηση, Έγκριση, κλπ), αποθηκεύουμε μόνο **επιχειρησιακές καταστάσεις**. Τα screens/actions/forms είναι transitions, όχι states.
Η αίτηση πρέπει να έχει ένα πεδίο `status` με πεπερασμένες τιμές, και κάθε αλλαγή να γίνεται μόνο μέσω επιτρεπτής μετάβασης, με atomic operation, audit log και έλεγχο δικαιωμάτων ανά ρόλο.

### 11.1 Status Enum
| Code | Ελληνική Ονομασία | Τύπος |
|------|-------------------|-------|
| DRAFT | Πρόχειρη αίτηση | Ενεργή |
| SUBMITTED | Υποβληθείσα αίτηση | Ενεργή |
| PENDING_PROTOCOL | Για πρωτόκολλο ΠΔΕΔΕ | Ενεργή |
| IN_REVIEW | Σε επεξεργασία από τμήμα αδειών | Ενεργή |
| WAITING_FOR_DOCUMENTS | Σε αναμονή δικαιολογητικών | Ενεργή |
| DECISION_PREPARATION | Ετοιμασία απόφασης | Ενεργή |
| PENDING_SIGNATURES | ΣΗΔΕ - προς υπογραφές | Ενεργή |
| COMPLETED | Ολοκληρώθηκε | Τελική |
| SUPERVISOR_REJECTED | Αρνητική έγκριση προϊσταμένου | Τελική |
| REJECTED_BY_LEAVES_DEPT | Απόρριψη από τμήμα αδειών | Τελική |
| CANCELLED_BY_APPLICANT | Ανάκληση από αιτούντα | Τελική |

### 11.2 Τι ΔΕΝ είναι states (είναι actions/screens/forms)
- Νέα Αίτηση
- Προεπισκόπηση Αίτησης
- Έγκριση από Προϊστάμενο
- Εγγραφή Αρ. Πρωτ. από ΣΗΔΕ
- Απαιτούνται ΔΚ/ΚΑ

### 11.3 Ρόλοι Workflow
| Ρόλος | Περιγραφή |
|-------|-----------|
| APPLICANT | Ο υπάλληλος που υποβάλλει την αίτηση |
| SUPERVISOR | Ο αρμόδιος προϊστάμενος που εγκρίνει ή απορρίπτει |
| LEAVES_HANDLER | Χειριστής αδειών (πρωτόκολλο, έλεγχος, απόφαση, ΣΗΔΕ) |
| ADMIN | Πλήρη διαχειριστικά δικαιώματα (χωρίς bypass του audit trail) |
| VIEW_ONLY | Read-only ρόλος |

| UNDER_PROCESSING | IN_REVIEW | ✅ Πιο σαφές |
| COMPLETED | COMPLETED | ✅ Ίδιο |
| REJECTED_OPERATOR | REJECTED_BY_LEAVES_DEPT | ✅ Ενοποίηση |

## PHASE 12: Dashboard Redesign - Κοινός Μηχανισμός Λίστας με Tabs/Filters/Sorting

### 12.1 Κοινές Αρχές

- **Query parameters**: `?sort=submitted_at&dir=asc&leave_type=annual&status=completed&page=2`
- **Filters**: GET form, διατηρούνται σε κάθε request
- **Pagination**: GET parameter, συνεργάζεται με sorting και filters
- **Role-aware actions**: Υπολογίζονται από status + role + permissions, όχι hardcoded

### 12.7 Κοινός Action Resolver
Η στήλη "Ενέργειες" παράγεται από κοινή συνάρτηση, όχι από if/else σε templates:
```python
def get_available_actions(leave_request, user):
    """Returns list of available actions based on status + role + permissions"""
    actions = []
    status = leave_request.status
    role = get_user_role(user, leave_request)
    
    # Action rules based on FSM transition matrix
    if status == 'DRAFT' and role == 'APPLICANT':
        actions.append(('view', 'ΠΡΟΒΟΛΗ'))
        actions.append(('edit', 'ΕΠΕΞΕΡΓΑΣΙΑ'))
        actions.append(('submit', 'ΥΠΟΒΟΛΗ'))
    elif status == 'SUBMITTED' and role == 'APPLICANT':
        actions.append(('view', 'ΠΡΟΒΟΛΗ'))
        if can_cancel(leave_request, user):
            actions.append(('cancel', 'ΑΝΑΚΛΗΣΗ'))
    # ... more rules
    return actions
```

## PHASE 13: KEDASY/SDEY Variant Architecture - Επεκτασιμότητα από την Αρχή

### Βασική Αρχή
Να υλοποιηθεί τώρα ο **κοινός πυρήνας** του συστήματος ως **γενικό workflow framework με παραμετροποίηση**, ώστε τα ΚΕΔΑΣΥ/ΣΔΕΥ να μπουν μετά ως **rules/configuration layer** και όχι ως "δεύτερο σύστημα".

### 13.1 Πότε είναι ασφαλές να το κάνεις σε 2 φάσεις
**Ασφαλές** αν τα ΚΕΔΑΣΥ/ΣΔΕΥ αλλάζουν μόνο:
- Κανόνες routing (ποιος εγκρίνει, ποια σειρά)
- Required attachments (ποια δικαιολογητικά)
- PDF templates/κείμενα (τι κείμενο απόφασης)
- Permissions/ορατότητα (ποιος βλέπει τι)

**ΜΗ ασφαλές** αν απαιτούν:
- Εντελώς διαφορετικά states
- Διαφορετικό lifecycle με επιπλέον στάδια
- Διαφορετική έννοια "προϊσταμένου" ή πολλαπλές εγκρίσεις
- Άλλο μοντέλο εγγράφων/αποφάσεων
- Άλλη δομή οργανωτικής μονάδας

## Notes & Decisions

- **KEDASY/SDEY Variant Architecture (Phase 13)**: Ο κοινός πυρήνας σχεδιάζεται ως γενικό workflow framework με παραμετροποίηση. Τα ΚΕΔΑΣΥ/ΣΔΕΥ θα μπουν ως rules/configuration layer σε δεύτερη φάση, όχι ως δομικές αλλαγές. LeaveType.workflow_variant = 'STANDARD' τώρα, 'KEDASY'/'SDEY' αργότερα.
- **Dashboard Redesign (Phase 12)**: Κοινός μηχανισμός λίστας με sortable columns, filters, pagination, role-aware actions. Tabs για προϊσταμένους και χειριστές. Query parameter preservation για sorting/filtering.
- **FSM Redesign (Phase 11)**: Καθαρές επιχειρησιακές καταστάσεις μόνο. Τα screens/actions/forms ΔΕΝ είναι states. State αλλάζει ΜΟΝΟ με explicit business action.
- **Status codes**: Keep English codes in database, change only display labels to Greek
- **Leave balance deduction**: BOTH check before submission AND deduct on completion
- **Status change on preview**: NOT automatic — status changes only when manager takes action (approve/reject), not on view/preview
- **SDEY Manager enforcement**: Override `can_request_leave` property — if user.department.department_type.code == 'SDEY' → return False. Enforce in view dispatch() and template rendering. Auto-set on registration approval.
- **Simple leaves attachments**: Attachments ARE allowed for holiday/verbal/educational leaves
- No email notifications (user decision)
- SSO deferred to later
- Password reset deferred to later
- Department model structure deferred
- Multiple roles handling deferred
- Reports/Excel export deferred
- DPO assignment deferred
- Self-registration exists with approval workflow
- Holidays on Sat/Sun don't count
- Data migration will be manual entry
- Default leave days: 25
- ESPA managers cannot request leave
- SDEY managers permanently cannot request leave
- Letterhead model needed for editable PDF header
- Two cancellation types: request cancellation (before processing) and leave cancellation (after completion)
- Only handler can delete requests in processing state
